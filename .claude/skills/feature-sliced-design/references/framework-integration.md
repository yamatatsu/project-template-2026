# Framework Integration

How to set up FSD within Vite + React, including directory placement,
routing integration with TanStack Router, and path alias configuration.

## General Principle

Place FSD layers inside `src/` to avoid naming conflicts with framework
directories. The FSD `app/` and `pages/` layers are conceptual layers, not
framework routing directories that happen to share the same names.

This project uses a single `@/*` path alias that maps to `src/*`, so imports
reference the layer path directly (`@/pages/home`, `@/shared/ui`,
`@/entities/user`). Configure the same alias in both `tsconfig.json` and the
Vite resolver so TypeScript and the bundler agree.

## Vite + React

### Directory structure

```text
my-vite-project/
  src/
    app/                   ← FSD app layer
      providers/
      router.tsx           ← Router definition (see TanStack Router below)
      styles/
      main.tsx             ← Entry point
    pages/
    shared/
  index.html
  vite.config.ts
  tsconfig.json
```

### Path aliases

Use a single `@/*` alias that maps to `src/*`. Configure it in
`tsconfig.json` and mirror it in `vite.config.ts` so the Vite resolver
agrees with TypeScript:

```json
// tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

```typescript
// vite.config.ts
import { fileURLToPath } from "node:url";
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      "@": fileURLToPath(new URL("./src", import.meta.url)),
    },
  },
});
```

Imports then reference the layer path directly: `@/pages/home`,
`@/shared/ui`, `@/entities/user`.

## TanStack Router

TanStack Router wires the application's routes. **Routing is the `app/`
layer's responsibility**, so the routes directory, the generated route tree,
and the router instance all live inside `app/`. Each route file is a thin
wrapper that points at a page's public API (`@/pages/*`).

### Where things go (the key FSD decision)

File-based routing wants a `routes/` directory and a generated
`routeTree.gen.ts`. FSD only recognizes the layer folders (`app/`, `pages/`,
…) at the `src/` top level, so a top-level `src/routes/` would be a non-FSD
folder. **Keep routing inside the `app/` layer**: configure the plugin so the
routes directory is `src/app/routes/` and the generated tree is
`src/app/routeTree.gen.ts`. Never create a top-level `src/routes/`.

- **Routes directory**: `app/routes/*.tsx`. Use **flat routes** (dot-separated
  filenames) so adding routes never bloats one file — this scales better than
  code-based routing as the route count grows.
- **Generated route tree**: `app/routeTree.gen.ts`. Do not hand-edit it.
  Commit it (so standalone `tsc` typecheck/CI works without a build step) but
  exclude it from your linter, formatter, and architecture linter (Steiger).
- **Router instance**: `app/router.tsx` binds the generated `routeTree` into
  `createRouter`. Keep it a thin layer; re-export `routeTree` from here so
  tests depend on `@/app/router`, not the generated file path.
- **Route shell** (header, side navigation, other layout chrome) goes on the
  root route `app/routes/__root.tsx`, which renders an app-layout component
  (compose `@/widgets/*` there), or move the shell into `widgets/` when it is
  a reusable composite block.
- **Keep route files thin.** A route file is wiring only: `createFileRoute`
  plus the page component from `@/pages/*`. Business logic, data fetching, and
  page UI live in the FSD `pages/` layer.

### Directory structure

```text
src/
  app/
    routes/                ← Flat route files (thin wrappers)
      __root.tsx           ← Root route → app-layout shell (renders <Outlet />)
      index.tsx            ← "/"
      profile.$id.tsx      ← "/profile/$id"
    routeTree.gen.ts       ← Generated by the plugin (do not edit; lint/format-exempt)
    router.tsx             ← createRouter(routeTree) + re-export routeTree
    app-layout.tsx         ← Layout chrome composed from @/widgets/*
    providers/
    main.tsx               ← Entry point: wires <RouterProvider />
  pages/
    home/
      ui/HomePage.tsx
      index.ts             ← export { HomePage } from './ui/HomePage'
    profile/
      ui/ProfilePage.tsx
      index.ts
  widgets/
  shared/
```

### Plugin configuration (Vite)

Point the TanStack Router plugin at the `app/` layer so nothing leaks to a
top-level `src/routes/`:

```typescript
// vite.config.ts
import { tanstackRouter } from "@tanstack/router-plugin/vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [
    // tanstackRouter must come before react.
    tanstackRouter({
      target: "react",
      routesDirectory: "./src/app/routes",
      generatedRouteTree: "./src/app/routeTree.gen.ts",
    }),
    react(),
  ],
});
```

### Flat route files (minimal example)

Flat routing maps dot-separated filenames to nested paths. A leaf that has
sibling sub-routes uses the `*.index.tsx` convention so you do **not** create
an intermediate layout file (which would force the parent to render an
`<Outlet />`).

```tsx
// src/app/routes/__root.tsx
import { createRootRoute } from "@tanstack/react-router";
import { AppLayout } from "../app-layout";

// Root = layout shell (providers + nav/header + <Outlet />).
export const Route = createRootRoute({ component: AppLayout });
```

```tsx
// src/app/routes/profile.$id.tsx
import { createFileRoute } from "@tanstack/react-router";
import { ProfilePage } from "@/pages/profile";

export const Route = createFileRoute("/profile/$id")({
  component: ProfilePage,
});
```

```tsx
// src/app/router.tsx
import { createRouter } from "@tanstack/react-router";
import { routeTree } from "./routeTree.gen";

export const router = createRouter({ routeTree });
export { routeTree }; // re-export so tests depend on @/app/router, not the generated file

declare module "@tanstack/react-router" {
  interface Register {
    router: typeof router;
  }
}
```

```tsx
// src/app/main.tsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { RouterProvider } from "@tanstack/react-router";
import { router } from "./router";

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <RouterProvider router={router} />
  </StrictMode>,
);
```

The page components (`ProfilePage`, …) come from `@/pages/*` public APIs. The
route files in `app/routes/` stay thin: they map paths to page components and
own no business logic.

> **Code-based routing** (defining the tree by hand in `app/router.tsx` with
> `createRoute`) is also valid FSD and avoids a generated file entirely.
> Prefer file-based flat routes when the route count is growing; prefer
> code-based when routes are few and you want zero generated artifacts.

> **`useParams({ from })` coupling**: page components that read typed params
> via `useParams({ from: "<route id>" })` must use the route id from the
> generated tree. With the `*.index.tsx` convention an index route id carries
> a trailing slash (e.g. `/tasks/$taskId/`), which differs from the
> code-based id (`/tasks/$taskId`).

## Key Reminders

1. **FSD lives in `src/`**: the FSD `app/` and `pages/` layers are
   architectural layers, not bundler routing folders.
2. **Route files are thin wrappers**: they import and render FSD page
   components. Business logic stays in FSD pages.
3. **Path aliases are required**: configure the single `@/*` → `src/*` alias
   in both `vite.config.ts` and `tsconfig.json`.
4. **Pages First still applies**: start with code in FSD `pages/` and extract
   only when needed.
5. **Routing stays in `app/`**: with file-based routing, point the plugin at
   `src/app/routes/` and `src/app/routeTree.gen.ts`. Never create a top-level
   `src/routes/` (it is not an FSD layer). Exclude the generated tree from
   lint/format/Steiger.
