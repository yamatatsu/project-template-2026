# 楽観ロック設計

このドキュメントは、本モノレポのミューテーションが**なぜこの形なのか**を人が読んで理解するための
設計書。実装は `apps/backend`（`wire/task.ts`・`repositories/task-db-repo.ts`）と `packages/db` の
`version` 列にあり、**このドキュメントとコードが食い違ったらコードが正**。日々の作業ルール（何を
すべきか）は [`apps/backend/CLAUDE.md`](../../apps/backend/CLAUDE.md)「楽観ロック」に置く。

**版はドメインが持たない**。`version` は業務ルールと直交する並行制御の関心なので、`Task` 型に載せず
repo が `Persisted<T>`（値と記録メタデータの対。`repositories/shared/index.ts`）として運ぶ（層分けの根拠は
[バックエンド設計](backend-architecture.md)「記録メタデータをドメイン型に持たせない理由」）。

## 概要

**更新（PUT）は `version` 列による楽観ロックを必須**とする（全テーブルが `version` を持つ。
スキーマ側の規約は [`packages/db/CLAUDE.md`](../../packages/db/CLAUDE.md)）。

### 中心不変条件 —— lost update を起こさない

> クライアントが読んだ版を土台にした更新だけを通し、その間に別の書き手が割り込んでいたら拒否する。

**削除（DELETE）には現状かけない**。テンプレート時点で削除の要求が定かでなく、複雑度・テスト量に
見合わないため。

## 版をどこで受けるか —— リクエストボディの `expectedVersion`

版は **`PUT` のリクエストボディのトップレベルフィールド `expectedVersion`（number）** で受ける。
**HTTP の条件付きリクエスト（`ETag` / `If-Match`）は使わない**。

### なぜ `If-Match` を使わないのか

`If-Match`（RFC 9110 §13.1.1）は「**サーバが以前に提供した** entity-tag」を送り返す仕組みで、`ETag`
レスポンスと対になって初めて成立する。`ETag` を出さずに `If-Match` だけ要求すると、クライアントは
サーバが一度も名乗っていないタグを自分で組み立てて送ることになる（`"<version>"` と引用符を手で
付ける羽目になる）——**HTTP の語彙を、その意味を成立させないまま借りている**状態になる。

では対にして `ETag` も返せばよいかというと、**この BFF では `ETag` の見返りを回収できない**。
`ETag` の本来の価値は HTTP キャッシュ機構（`If-None-Match` → 304、中間プロキシや CDN が代理で条件を
判断する）にある。しかしここは特定フロント専用の BFF で、第三者クライアントも中間キャッシュもいない
（認証は HttpOnly セッション Cookie なので、そもそも共有キャッシュに乗らない）。

一方でコストは実在する:

1. **版の読み取り側が型安全から外れる**。Hono RPC の `InferResponseType` は**レスポンスヘッダを型に
   載せない**。版をヘッダで返すと、フロントは `res.headers.get('etag')` を手読みし、キャッシュへ自力で
   持ち回ることになる。このリポの中心的な価値（エンドツーエンドの型安全）を、**版という最も落として
   はいけない値**で捨てることになる。
2. **一覧が版を運べない**。`ETag` は 1 レスポンスに 1 個なので、`GET /tasks` の各行の版を表現できない。
   一覧から直接編集に入れず、編集前に必ず詳細 `GET` を挟む羽目になる。

**回収できない標準に払うコストは純損**。半端に借りるくらいなら借りない、というのがこの設計の立場。
版は HTTP の precondition ではなく**リソースの属性**として body で往復させる。

### 得られるもの

- **往復の両方向が RPC の型に載る**。リクエストの `expectedVersion` は `InferRequestType` に、
  レスポンスの `meta.version` は `InferResponseType` に現れ、**送信も読み取りも型で強制される**。
- 一覧が各行の版を自然に運ぶ（`meta.version`）。一覧から直接編集に入れる。
- 版を受けるための middleware が要らない。**ステータスの対応づけが 400 / 404 / 409 の 3 つで済む**。

### 命名 —— リクエストは `expectedVersion`、レスポンスは `meta.version`

**往復で名前を変える**ことで、これがリソースの内容ではなく**クライアントが編集の土台にした前提**で
あり、**リクエストにしか存在しない**ことを表現する（レスポンスに `expectedVersion` は現れない。返るのは
サーバが決めた現在版 `meta.version`）。`currentVersion` は「今 DB にある版」と読み違えられ、CAS の条件を
逆に書く事故を招くので使わない。

### スキーマ

| スキーマ                     | 使うルート       | 中身                                          |
| ---------------------------- | ---------------- | --------------------------------------------- |
| `taskInputSchema`            | `POST /tasks`    | 内容フィールドのみ                            |
| `conditionalTaskInputSchema` | `PUT /tasks/:id` | `taskInputSchema.extend({ expectedVersion })` |

「条件付きリクエスト」は境界（HTTP）の概念なので、wire の命名規約（`create`/`update` という**コマンド層の
語彙**を wire に持ち込まない）を破らずに済む。内容部分の定義源は `taskInputSchema` 1 つに保ち、PUT 用は
`.extend` で派生させる。

`PUT /tasks/:id` は **PATCH ではなく PUT（全体置換）**。全フィールド必須なので送られたボディが
リソースの全体像そのものになり、送値どおりに反映する（旧値は保持しない）。

### 引き受けたトレードオフ

- **DELETE に楽観ロックを掛けたくなったとき body が使えない**（DELETE のボディは仕様上曖昧で、プロキシに
  落とされることがある）。そのときはクエリパラメータで受けるか、DELETE だけヘッダに戻す判断が要る。
  現状 DELETE には掛けないので、いま払う代償ではない。
- **PUT は全体置換なのに、置換対象でない値（前提）が body に混ざる**。`expectedVersion` という名前と、
  レスポンスに現れないことでこのねじれを表現する。
- **POST と PUT の入力形が完全一致でなくなる**（`expectedVersion` の 1 フィールド差）。

## ステータスの対応づけ

1. `expectedVersion` の欠如・形式不正 —— **400**（`zValidator('json')`。他の必須フィールド欠落と同じ扱い）。
2. `findTask` で存在確認 —— 無ければ **404**。
3. `saveTask` の**基底版を条件にした CAS** —— クライアントが土台にした版と DB の現在版の一致を判定する
   **唯一の点**。stale な版も、load→save 間の割り込みも、同じ `WHERE version = expectedVersion` が原子的に
   落とす。1 件も更新できなければ **409**。

メモリ内の事前チェック（ロードした版と `expectedVersion` の突き合わせ）は**置かない** —— CAS の完全な
部分集合であり、同じ競合を 2 か所で判定すると 409 の分岐が増えるだけで正しさは変わらない。負けると
分かっている UPDATE を 1 回投げるコストは、競合が稀な前提では誤差。

### 409 であって 412 / 428 ではない理由

**412 Precondition Failed は使わない**。412 は「リクエスト**ヘッダフィールド**で与えられた 1 つ以上の条件が
false と評価された」（RFC 9110 §15.5.13）を意味し、ヘッダ条件を一切使わない以上あたらない。**409 Conflict**
は「対象リソースの現在の状態との衝突によりリクエストを完了できなかった」（§15.5.10）で、lost update
そのもの。

**428 Precondition Required も使わない**。precondition という HTTP の概念自体を使わないので、版の欠落は
「必須フィールドが無い」ただの 400 でよい。

### 版の決定は repo に閉じる

初期値（`addTask` の 1）も増分（`saveTask` の `incrementVersion` ＝ DB 側の `version + 1`）も repo が
決める —— 版は永続化の関心なので、その増分規約も永続化側の事実。ドメイン関数（`createTask`／
`applyUpdate`）は版に触れない。増分を書けるのは CAS を通った書き込みだけなので、版の単調増加が構造的に
保たれる。増分片 `incrementVersion` は `repositories/shared/index.ts` に置き、全テーブルの repo で共有する
（`version + 1` は CAS でマッチした行では `expectedVersion + 1` に一致する）。テーブルを丸ごと受けて
helper が `version` 列を採るので、版列を持たないテーブルや別列の誤りを型で弾ける。

## 409 のレスポンスと回復

409 の body は **`{ error: 'Version conflict', entity: 'task', id }` のみ**で、現在版は返さない。
競合は稀な前提で、フロントは読み込んだ `meta.version` を `expectedVersion` として送り、409 を受けたら
対象を再取得して再送信する。
